# `role` 与 `enabled` 说明

更新时间：2026-05-08

## 1. 先说结论

在“场景语义配置区”里，字段最核心的三个配置是：

- `semantic_name`：这个字段在业务上叫什么
- `role`：这个字段在分析里扮演什么角色
- `enabled`：这个字段现在允不允许进入后续分析流程

一句话：

- `semantic_name` 决定“业务上怎么叫”
- `role` 决定“分析时怎么用”
- `enabled` 决定“现在能不能用”

---

## 2. `role` 到底是什么

`role` 是在告诉系统：

**这个字段在分析里，到底是拿来算、拿来分组、拿来看时间趋势，还是拿来做筛选。**

最常见的四种：

### 2.1 `metric`

- 含义：指标
- 作用：拿来统计、求和、求平均、求最大最小
- 常见例子：
  - 销售额
  - 价格
  - 销量

最粗暴的理解：

- `metric = 算什么数`

### 2.2 `dimension`

- 含义：维度
- 作用：拿来分组、拆分结果
- 常见例子：
  - 品牌
  - 类目
  - 颜色

最粗暴的理解：

- `dimension = 按什么看`

### 2.3 `time`

- 含义：时间字段
- 作用：拿来做时间趋势分析
- 常见例子：
  - 上新时间
  - 创建时间
  - 抓取日期

最粗暴的理解：

- `time = 按时间怎么看`

它本质上也是一种维度，但系统会把它当成“时间分析字段”单独理解。

### 2.4 `filter`

- 含义：过滤条件
- 作用：拿来限制范围
- 常见例子：
  - 性别
  - 场景
  - 是否新品

最粗暴的理解：

- `filter = 限定什么范围`

---

## 3. 为什么必须定义 `role`

因为如果不定义，系统就容易把字段用错。

例如：

- `品牌` 如果被当成 `metric`，系统可能会去“统计平均品牌”，这是错误的。
- `价格` 如果被当成 `dimension`，系统可能会按每个价格值分组，结果会非常碎。
- `上新时间` 如果没有标成 `time`，系统就不容易理解你想看趋势。
- `性别` 如果被当成主维度，而不是过滤条件，分析方向可能跑偏。

所以 `role` 的本质作用不是“备注”。

它是：

**防止系统把字段用错。**

---

## 4. `enabled` 是什么

`enabled` 是字段开关。

只有两种状态：

- `enabled = true`
- `enabled = false`

含义很简单：

### 4.1 `enabled = true`

- 这个字段允许进入后续分析流程
- 查询计划可以选它
- SQL 生成可以用它

### 4.2 `enabled = false`

- 这个字段虽然还在配置里
- 但暂时不参与后续分析
- 相当于“先留着，但不上场”

最简单的理解：

- `enabled = 这个字段能不能上场`

---

## 5. 最简单的例子

### 5.1 按品牌看平均价格

配置应该是：

- `品牌` -> `dimension`
- `价格` -> `metric`

因为：

- 品牌是拿来分组的
- 价格是拿来计算平均值的

### 5.2 看最近 30 天上新趋势

配置应该是：

- `上新时间` -> `time`
- `价格` 或 `商品数` -> `metric`

因为：

- 时间是趋势轴
- 指标是趋势值

### 5.3 只看女装里的品牌分布

配置应该是：

- `性别` -> `filter`
- `品牌` -> `dimension`

因为：

- 性别是限制分析范围
- 品牌是展示和分组维度

---

## 6. 判断一个字段该配成什么 `role`

可以用一个非常简单的方法判断：

### 第一步：问自己

这个字段是：

- 拿来算数的吗？
- 拿来分组的吗？
- 拿来看时间趋势的吗？
- 拿来筛选范围的吗？

### 第二步：按下面规则配

- 拿来算数：`metric`
- 拿来分组：`dimension`
- 拿来看时间：`time`
- 拿来筛选：`filter`

### 第三步：如果拿不准

优先这样处理：

- 数值类，且经常被汇总：`metric`
- 文本类标签、分类字段：`dimension`
- 日期时间类：`time`
- 只用来限制条件、不太展示：`filter`

---

## 7. `role` 和 `enabled` 后续怎么影响流程

后续链路里，这两个字段都是真正会被用上的。

### 7.1 查询计划阶段

系统会先从启用字段里选候选字段：

- `metric` 字段进入指标候选
- `dimension/time` 字段进入维度候选
- `enabled = false` 的字段不会参与

也就是说：

- `role` 决定字段归到哪一类
- `enabled` 决定字段是否有资格参与

### 7.2 SQL 生成阶段

SQL 不直接使用 `role` 来写语法，
但前面的查询计划已经因为 `role` 不同，选出了不同的字段组合。

所以 `role` 是通过“前置筛选逻辑”影响 SQL 的。

### 7.3 LLM / SQL 结果 Agent 阶段

LLM 在理解这个场景时，也会看到字段角色和启用状态。

所以：

- `role` 会影响 LLM 对字段用途的理解
- `enabled` 会影响哪些字段被当作可用字段交给后续流程

---

## 8. 你真正要记住的

只记下面三句就够了：

- `semantic_name`：业务上叫什么
- `role`：分析时怎么用
- `enabled`：现在能不能用

其中：

- `role` 解决“字段用法”
- `enabled` 解决“字段是否上场”

---

## 9. 最后再用一句话总结

`role` 不是备注，
它是在告诉系统：

**这个字段到底是指标、维度、时间，还是过滤条件。**

`enabled` 也不是装饰开关，
它是在告诉系统：

**这个字段现在是否允许进入后续分析流程。**
