## 2. 当前业务框架

当前系统适合按以下流程实现业务问题：

1. 业务目标定义  
   明确问题要回答什么经营决策，例如价格定位、上新节奏、品类结构、竞品差异。

2. 场景选择  
   选择对应场景：商品价格分析、竞品与价格分析、商品结构分析、趋势与爆款分析。

3. 语义字段建模  
   把业务词映射到真实字段，例如品牌 -> `clothing_info.BrandName`，价格 -> `clothing_info.Price`。

4. 预设问题生成  
   每个预设问题必须绑定：
   - 可用字段
   - 指标公式
   - 过滤条件
   - 输出粒度
   - SQL 验收口径

5. SQL 生成与执行  
   只允许生成 MySQL 可执行只读 SQL。字段引用必须使用反引号，不能使用 PostgreSQL 风格双引号。

6. 数据质量与业务还原校验  
   检查 SQL 是否回答原业务问题，而不是只检查是否有结果。

7. 汇报产物生成  
   只有通过业务还原校验的数据，才进入 slide/deck/PPT 生成。

## 3. 预设问题重规划原则

现有前端预设问题偏通用，例如“统计商品总数”“按 BrandName 统计商品数”。这些能测试 SQL 链路，但不能代表业务分析。

新的预设问题必须符合三条规则：

1. 不能提出当前数据无法回答的问题  
   例如“不同平台销售价差”在没有有效平台字段前不能作为默认问题。

2. 问题必须明确输出指标  
   例如“均价、SKU 数、最高价、最低价、价差、价格带占比”。

3. 问题必须明确业务粒度  
   例如品牌、品类、场景、材质、功能、日期。

## 5. 平台价差问题的 0-1 实现路径

业务问题：“分析各品牌在不同平台的销售价差异并生成汇报”

当前状态：不满足。原因是没有有效的“平台”字段。

### 5.1 第 0 步：定义平台口径

必须先确定平台到底指什么：

| 口径 | 可行性 | 说明 |
| --- | --- | --- |
| 销售平台 | 当前不可行 | 需要真实平台字段，如天猫、京东、抖音、官网等 |
| 店铺 | 当前不可行 | `ShopName` 为空，`ShopId` 单值 |
| 来源站点域名 | 可弱实现 | 从 `SourceUrl` 派生 `dot-st`、`goldwin`，但这是来源站点，不是严格销售平台 |
| 穿着/使用场景 | 可实现但不能叫平台 | 来自 `clothing_scene_info.Scene` |

### 5.2 第 1 步：补数据字段

推荐新增平台维度表：

```sql
CREATE TABLE platform_dim (
  platform_id VARCHAR(64) PRIMARY KEY,
  platform_name VARCHAR(128) NOT NULL,
  platform_type VARCHAR(64) NOT NULL,
  source_domain VARCHAR(255) NOT NULL
);
```

推荐新增商品平台映射：

```sql
CREATE TABLE clothing_platform_info (
  clothing_id BIGINT NOT NULL,
  platform_id VARCHAR(64) NOT NULL,
  source_url VARCHAR(1000),
  captured_at DATETIME,
  PRIMARY KEY (clothing_id, platform_id)
);
```

如果短期不能补表，可先建派生视图：

```sql
CREATE OR REPLACE VIEW clothing_platform_view AS
SELECT
  Id AS ClothingId,
  CASE
    WHEN SourceUrl LIKE '%dot-st.com%' THEN 'dot-st'
    WHEN SourceUrl LIKE '%goldwin.co.jp%' THEN 'goldwin'
    ELSE 'unknown'
  END AS platform_name,
  SourceUrl
FROM clothing_info
WHERE SourceUrl IS NOT NULL AND SourceUrl <> '';
```

注意：这个视图只能支持“来源站点价差”，不能宣称是完整平台价差。

### 5.3 第 2 步：更新语义场景

在“商品价格分析”和“竞品与价格分析”中新增字段：

| 语义名 | 表 | 字段 | role | 说明 |
| --- | --- | --- | --- | --- |
| 平台 | `clothing_platform_view` 或 `clothing_platform_info` | `platform_name` | dimension | 平台/来源站点维度 |
| 商品ID | `clothing_platform_view` | `ClothingId` | filter | 用于关联 |

新增关系：

| left | right | join |
| --- | --- | --- |
| `clothing_info.Id` | `clothing_platform_view.ClothingId` | INNER |

### 5.4 第 3 步：重写平台价差预设问题

平台数据补齐后，预设问题应改成：

1. 各品牌在不同平台的 SKU 数、平均销售价、最低价、最高价是多少？
2. 各品牌的平台间平均价差和价差率是多少，按价差率降序返回前 20？
3. 每个品牌的最高价平台和最低价平台分别是什么？
4. 在同一二级类目下，各品牌跨平台价差是否显著？
5. 平台维度下高价带 SKU 占比是否存在差异？

### 5.5 第 4 步：SQL 目标模板

平台价差的目标 SQL 应至少包含：

```sql
WITH platform_price AS (
  SELECT
    ci.BrandName AS `品牌`,
    pv.platform_name AS `平台`,
    COUNT(DISTINCT ci.Id) AS `SKU数`,
    AVG(ci.Price) AS `平均价格`,
    MIN(ci.Price) AS `最低价格`,
    MAX(ci.Price) AS `最高价格`
  FROM clothing_info ci
  INNER JOIN clothing_platform_view pv
    ON ci.Id = pv.ClothingId
  WHERE ci.Price IS NOT NULL
  GROUP BY ci.BrandName, pv.platform_name
),
brand_gap AS (
  SELECT
    `品牌`,
    MAX(`平均价格`) - MIN(`平均价格`) AS `平台均价差`,
    CASE
      WHEN MIN(`平均价格`) = 0 THEN NULL
      ELSE (MAX(`平均价格`) - MIN(`平均价格`)) / MIN(`平均价格`)
    END AS `平台价差率`,
    COUNT(DISTINCT `平台`) AS `平台数`
  FROM platform_price
  GROUP BY `品牌`
)
SELECT
  p.*,
  g.`平台均价差`,
  g.`平台价差率`,
  g.`平台数`
FROM platform_price p
INNER JOIN brand_gap g
  ON p.`品牌` = g.`品牌`
WHERE g.`平台数` >= 2
ORDER BY g.`平台价差率` DESC, p.`品牌`, p.`平台`;
```

验收条件：

| 条件 | 要求 |
| --- | --- |
| SQL 可执行 | MySQL 8.0 可执行，只用反引号 |
| 平台字段 | 必须来自平台字段/视图，不能来自 `Scene` |
| 平台数 | 品牌至少覆盖 2 个平台才计算价差 |
| 指标 | 必须包含均价、价差、价差率、SKU 数 |
| 样本保护 | 样本数过小需标记，不直接下强结论 |

## 6. 不同业务问题的 0-1 实现方式

### 6.1 商品规模盘点

目标：确认数据量、品牌覆盖、品类覆盖。

实现步骤：

1. 选择场景：商品结构分析。
2. 语义字段：品牌、一级类目、二级类目、叶子类目。
3. SQL 指标：`COUNT(DISTINCT Id)`、`COUNT(DISTINCT BrandName)`、`COUNT(DISTINCT Category)`。
4. 输出：总 SKU、品牌数、品类数、Top 品牌、Top 品类。
5. 验收：结果行数大于 0，字段均来自 `clothing_info`。

### 6.2 品牌价格定位

目标：识别品牌价格层级和价格跨度。

实现步骤：

1. 选择场景：商品价格分析。
2. 语义字段：品牌、价格。
3. SQL 指标：SKU 数、平均价、最低价、最高价、价格跨度。
4. 输出：品牌价格排行、价格跨度排行。
5. 验收：每个品牌包含 SKU 数，样本数小于 3 的品牌标记为低可信。

### 6.3 品类价格结构

目标：看不同品类的价格带和价格水平。

实现步骤：

1. 选择场景：商品价格分析。
2. 语义字段：一级类目、二级类目、价格。
3. SQL 指标：SKU 数、平均价、价格带分布。
4. 输出：品类价格结构表、价格带占比图。
5. 验收：价格带口径固定，不能由模型随意改。

### 6.4 品牌竞品对比

目标：比较品牌在同一品类下的定位差异。

实现步骤：

1. 选择场景：竞品与价格分析。
2. 输入品牌列表和品类范围。
3. SQL 过滤：`BrandName IN (...)`，`SubCategory = ...`。
4. SQL 指标：SKU 数、均价、最低价、最高价、功能覆盖、材质覆盖。
5. 输出：品牌对比矩阵。
6. 验收：必须同品类比较，不能跨品类直接下结论。

### 6.5 商品结构分析

目标：看品牌的 SKU 布局、品类覆盖、颜色覆盖。

实现步骤：

1. 选择场景：商品结构分析。
2. 语义字段：品牌、一级类目、二级类目、叶子类目、颜色。
3. SQL 指标：SKU 数、覆盖类目数、覆盖颜色数、占比。
4. 输出：品牌结构排行、品类覆盖矩阵。
5. 验收：占比必须说明分母，是全量占比还是品牌内占比。

### 6.6 上新趋势分析

目标：识别最近上新节奏和高峰。

实现步骤：

1. 选择场景：趋势与爆款分析。
2. 语义字段：上架时间、抓取日期、品牌、品类。
3. SQL 指标：每日新增 SKU、品牌新增 SKU、品类新增 SKU。
4. 输出：趋势线、Top 品牌、Top 品类。
5. 验收：当前数据时间跨度有限，不能宣称长期趋势。

### 6.7 场景/功能/材质洞察

目标：分析商品适用场景、功能卖点、材质使用差异。

实现步骤：

1. 选择场景：竞品与价格分析或趋势与爆款分析。
2. 关联表：`clothing_scene_info`、`clothing_functions_info`、`clothing_fiber_info`。
3. SQL 指标：SKU 数、品牌数、均价。
4. 输出：场景/功能/材质分布。
5. 验收：多标签表会导致一件商品多行，必须用 `COUNT(DISTINCT ci.Id)` 做 SKU 数。

## 7. SQL Agent 必须增加的约束

为了避免新模型生成“看起来对、实际不对”的 SQL，需要给 SQL Agent 增加硬约束：

1. MySQL 方言约束  
   - 字段别名用反引号：`` AS `品牌` ``
   - 禁止 PostgreSQL 双引号：`"品牌"`

2. 平台字段约束  
   - 如果语义字段中没有“平台”，必须返回风险提示，不允许把 `Scene` 当平台。
   - 如果用户问题包含“平台”，但没有平台字段，应要求补平台字段或改成“来源站点/场景”。

3. 指标完整性约束  
   - 价差问题必须包含均价、最低价、最高价、价差或价差率。
   - 排名问题必须包含排序指标。
   - 趋势问题必须包含时间字段和聚合周期。

4. 样本量保护  
   - 所有聚合结果都输出 `SKU数`。
   - SKU 数小于 3 的结果标记为低样本。

5. 业务还原校验  
   - SQL 执行成功不等于业务达成。
   - 必须校验业务词是否映射到正确字段。

## 9. 实施优先级

### P0：立即修正

1. 前端预设问题从“通用 SQL 测试题”改为“业务问题”。
2. SQL Agent 增加 MySQL 方言硬约束，禁止双引号别名。
3. SQL Agent 增加“平台字段缺失”拦截，不能把场景标签当平台。
4. 所有聚合 SQL 默认输出 `SKU数`。

### P1：一周内补齐

1. 新增来源站点派生字段或视图。
2. 在语义字段中加入“来源站点”，并明确它不是销售平台。
3. 增加业务还原检查：业务词 -> 语义字段 -> SQL 字段。
4. 把失败 SQL 的真实 MySQL error 返回到前端，方便定位。

### P2：平台价差正式可用

1. 接入真实平台维表或平台字段。
2. 建立商品与平台映射。
3. 建立平台价差 SQL 模板。
4. 汇报产物中区分：
   - 平台价差
   - 来源站点价差
   - 场景价格差异

## 10. 最终判断

当前系统已经能真实接入数据库，并能完成商品价格、品牌、品类、结构、趋势、场景、功能、材质等分析。

但“不同平台销售价差”这类问题还没有达到业务目标，因为当前数据没有有效平台维度。正确做法不是让模型继续猜，而是先补平台语义或改问题口径。

因此，从 0-1 的正确路线是：

1. 先按当前可用字段上线商品价格、结构、趋势、竞品、场景功能分析。
2. 同时把“平台价差”标记为需补数据的问题，不作为默认预设问题。
3. 补平台维表/来源站点视图后，再上线平台价差类问题。
4. 所有问题进入汇报产物前必须通过“业务还原校验”。
