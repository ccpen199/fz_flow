# 场景语义配置与 LLM 推荐说明

更新时间：2026-04-13

## 1. 先说结论（你这次关心的点）

- `LLM 推荐`：用于生成候选字段/关系（可勾选）。
- `导入`：把勾选的候选写入场景定义，并同步进语义缓存。
- `语义配置区`：统一维护“已选字段 + 已选关系”。
- `可执行集合`：等于语义缓存里 `enabled=true` 的字段，查询链路只读该集合。

当前版本已简化为单工作区，不再在 UI 中区分 `modeled/effective`。

## 2. 页面上每块到底干什么

## 2.1 场景选择区

- 选择当前分析场景（scene）。
- 场景决定后续可用字段、关系、推荐候选范围、会话默认语义上下文。

## 2.2 语义配置区（底部）

- 这是“语义层管理区”，不是报表展示区。
- 主要管理：
  - 字段语义（`semantic_name`、`role`、`aggregation`、`unit`、`aliases`）
  - 字段启停（`enabled`）
  - 关系配置（join 关系）

为什么要有这块：
- 避免 LLM 直接面对原始库表导致不稳定。
- 把“可分析字段”与“原始字段”隔离，保证可控、可复用、可审计。

## 2.3 LLM 推荐区

- 根据当前场景 + 目标（goal）推荐候选 `tables/fields/relations`。
- 你可以勾选/取消勾选后再导入，避免一键全量污染语义层。

## 2.4 可执行集合

- 可执行集合本质是语义缓存中 `enabled=true` 的字段集合。
- 执行链路（生成查询计划、执行查询）读的是这套可执行语义，而不是所有候选。

## 2.5 append / replace 的作用

- `append（追加）`：把本次勾选的候选字段/关系追加到当前场景，不清空已有配置；已有重复项会尽量跳过。
- `replace（替换）`：先清空当前场景已有字段/关系，再导入本次勾选的候选；同时会替换当前场景的有效语义字段缓存。

建议：

- 日常调试优先用 `append`。
- 只有确认要重建当前场景口径时才用 `replace`。
- `replace` 不是“撤销某一个字段/关系”，而是“用当前勾选集合重建场景”。

## 3. 全流程（从选场景到执行）

1. 选择场景  
系统加载场景定义（scene fields/relations）和语义缓存（统一列表）。

2. 点击 LLM 推荐  
生成候选字段/关系，默认可勾选后人工筛选。

3. 点击导入  
将勾选候选导入：
- 写入场景定义（scene fields/relations）
- 同步写入语义缓存（统一列表）

4. 在语义配置区确认  
检查字段语义、角色、启停状态、关系是否符合业务口径。

5. 创建会话并执行  
查询计划和执行阶段读取“可执行语义字段”，产出查询结果与后续内容。

## 4. 本次修复说明

问题现象：
- 勾选导入后，LLM 推荐没有进入可执行集合。

根因：
- 之前导入只更新了 `scene.fields/scene.relations`，但执行链路读取的是 `semantic-cache`，两套数据未同步。

修复内容：
- 导入接口新增语义缓存同步逻辑：
  - 勾选字段导入后写入 `semantic-cache` 统一列表。
  - `replace` 模式下清空场景旧缓存后再写入新勾选项。

影响：
- 导入后会立即在语义配置区看到字段，并进入可执行集合。

## 5. 使用建议（下次可按这个做）

1. 先选场景，再点推荐，不要跨场景复用老候选。
2. 推荐后先“批量取消”，只勾选你确认要进执行链路的字段/关系。
3. 首次建议用 `append`，确认无误再用 `replace` 做收敛。
4. 导入后在语义配置区检查 `enabled` 和 `role`，再创建会话执行。
5. 字段可在“字段语义”工作区编辑、禁用或删除；关系可在“ER 关系”工作区删除。
