# 字段与 SQL 阶段实现对照评估报告

- 评估日期：2026-04-14
- 评估范围：`apps/vibe_backend`、`integrations`
- 对照基准：PRD 中 DeepFlow 集成、上下文记忆、模型调用链路、场景配置与验收项

## 一、执行结论

当前系统已达到“可运行原型”阶段（字段 + SQL 端到端基本可用），但尚未达到 PRD 目标中的“场景化、可解释、可持续多轮、可恢复”产品形态。  
核心结构性差距集中在三点：`DeepFlow 未进入主链`、`上下文记忆未参与决策`、`模型调用治理不足`。

| 类别 | 状态 | 结论 |
|---|---|---|

## 四、优先级整改建议

### P0（必须先做）

1. DeepFlow 并入主执行链：让 `analysis_sessions -> planner -> tool/sql` 统一走可观测 agent runtime。
2. 记忆注入决策：把会话历史、偏好、上轮结论注入 SQL 规划与重写阶段。
3. 默认开启可恢复持久化：至少在生产/预发默认 `persist on`。

### P1（上线前必须补齐）

1. SQL 治理：场景白名单（表/字段）、`LIMIT` 强制、超时、扫描/返回行数上限、审计日志。
2. 任务闭环：产出结构化 `task_list/completion_criteria/next_goal` 并驱动下一步推荐。
3. 场景规则配置化：playbook、字段口径、ER、校验规则支持管理端维护。

### P2（体验与可信度增强）

1. Slide 页面级溯源：每页输出“结论-指标-字段-SQL-时间范围”链路。
2. 失败恢复与重试策略：中断后可从 `plan/run` 级别续跑。

## 五、里程碑建议（可直接作为执行计划）

1. M1（3-5 天）：打通 DeepFlow 主链 + 记忆注入 + 持久化默认开启。
2. M2（3-5 天）：补齐 SQL 治理与任务闭环结构。
3. M3（2-4 天）：完成配置化场景规则与 slide 级可解释链路。

## 六、验收口径（建议）

1. 同一会话连续三轮追问，SQL 计划可利用前序上下文。
2. 服务重启后会话可恢复并继续追加。
3. 任意执行均可回溯到场景、字段、SQL 与结论映射。
4. 非白名单字段/表无法被执行；超时与行数限制生效并可审计。
