# 项目交接：数据库导入与启动配置说明

本文用于把当前 `fz_workflow` 项目复制给其他人后，从 0 启动同一套本地流程。

## 1. 交接范围

当前交接需要同时复制：

- 项目代码目录：`fz_workflow/`
- 当前完整数据库 SQL：`docs/Sql/dataservice_test_local_current_2026-05-15.sql`
- 后端配置文件：`deploy/fz-workflow-demo.app.env`
- 启动脚本：`scripts/start_vibe_stack.sh`、`scripts/start_vibe_backend.sh`、`scripts/start_vibe_frontend.sh`
- 可选 PPT/Presenton 目录：`third_party/presenton/`

不要只复制原始业务 dump，否则配置中心、场景字段、语义缓存、历史记录会缺失。

## 2. 当前数据库 SQL

推荐交接使用：

```text
docs/Sql/dataservice_test_local_current_2026-05-15.sql
```

这份 SQL 是当前本地库 `dataservice_test_local` 的完整导出，包含：

- 业务数据表：`clothing_info`、`clothing_fiber_info`、`clothing_functions_info`、`clothing_images_color`、`clothing_pattern_info`、`clothing_scene_info`、`clothing_texture_info`
- 配置中心表：`vibe_scene`、`vibe_scene_field`、`vibe_scene_relation`、`vibe_semantic_field_cache`
- 运行状态/历史表：`vibe_query_run_state`、`vibe_workflow_state`、`vibe_workflow_stage_state`、`llm_agent_drafts`

原始业务 dump 仍保留在：

```text
docs/Sql/dataservice_test(1).sql
```

它只适合重新导入原始业务表，不包含当前完整配置中心状态。

## 3. 重新导出当前数据库

如果后续又修改了配置中心、预设问题、执行历史，重新交接前应重新导出数据库。

标准导出命令：

```bash
mysqldump -h127.0.0.1 -P3306 -uroot -p \
  --default-character-set=utf8mb4 \
  --single-transaction \
  --routines \
  --triggers \
  --databases dataservice_test_local \
  > docs/Sql/dataservice_test_local_current_$(date +%F).sql
```

如果希望导出的 SQL 和当前交接文件一样直接固定到 `dataservice_test_local`，可以导出后检查文件头部是否包含：

```sql
CREATE DATABASE IF NOT EXISTS `dataservice_test_local` CHARACTER SET utf8mb4 COLLATE utf8mb4_bin;
USE `dataservice_test_local`;
```

本机如果 `mysqldump` 遇到 MySQL 认证插件问题，可以换用数据库客户端导出，或先升级本机 MySQL client。

## 4. 环境要求

建议环境：

- macOS/Linux
- Python `3.12+`
- MySQL `8.x`
- 已安装 `pip`
- 可选：Node.js，用于启动 Presenton Next.js 前端

后端 Python 依赖来自：

```text
apps/vibe_backend/pyproject.toml
```

## 5. 导入数据库

先创建数据库，再导入当前完整 SQL：

```bash
mysql -h127.0.0.1 -P3306 -uroot -p \
  -e "CREATE DATABASE IF NOT EXISTS dataservice_test_local CHARACTER SET utf8mb4 COLLATE utf8mb4_bin;"

mysql -h127.0.0.1 -P3306 -uroot -p dataservice_test_local \
  < docs/Sql/dataservice_test_local_current_2026-05-15.sql
```

如果 MySQL 客户端遇到本机认证插件问题，可以使用 Navicat、DataGrip、DBeaver 等客户端导入同一份 SQL。

如果只需要导入原始业务 dump，可以使用项目内脚本：

```bash
.venv/bin/python scripts/import_dataservice_mysql.py \
  --host 127.0.0.1 \
  --port 3306 \
  --user root \
  --password root \
  --database dataservice_test_local \
  --sql-file 'docs/Sql/dataservice_test(1).sql'
```

注意：这个脚本默认面向原始业务 dump，并会追加 `scripts/mysql_index_patch.sql` 索引补丁；交接复刻当前状态时优先导入 `dataservice_test_local_current_2026-05-15.sql`。

导入后可检查：

```bash
mysql -h127.0.0.1 -P3306 -uroot -p dataservice_test_local \
  -e "SHOW TABLES; SELECT COUNT(*) AS clothing_count FROM clothing_info; SELECT COUNT(*) AS scene_count FROM vibe_scene;"
```

## 6. 数据库账号密码配置

后端通过环境变量读取 MySQL 配置，默认值在代码中是：

```text
MYSQL_HOST=127.0.0.1
MYSQL_PORT=3306
MYSQL_USER=root
MYSQL_PASSWORD=root
MYSQL_DATABASE=dataservice_test_local
```

读取这些变量的后端模块包括：

- `apps/vibe_backend/app/services/query_service.py`
- `apps/vibe_backend/app/services/sql_result_agent_service.py`
- `apps/vibe_backend/app/services/llm_agent_service.py`
- `apps/vibe_backend/app/services/runtime_persistence_service.py`
- `apps/vibe_backend/app/services/scene_cache_service.py`
- `apps/vibe_backend/app/services/semantic_field_cache_service.py`
- `apps/vibe_backend/app/routers/scenes.py`
- `apps/vibe_backend/app/routers/clothing.py`

推荐把数据库配置写入：

```text
deploy/fz-workflow-demo.app.env
```

示例：

```env
MYSQL_HOST=127.0.0.1
MYSQL_PORT=3306
MYSQL_USER=root
MYSQL_PASSWORD=your_mysql_password
MYSQL_DATABASE=dataservice_test_local
```

也可以启动前临时注入：

```bash
MYSQL_PASSWORD=your_mysql_password bash scripts/start_vibe_stack.sh
```

## 7. 大模型配置

后端默认读取：

```text
deploy/fz-workflow-demo.app.env
```

核心配置项：

```env
LLM_AGENT_PROVIDER=modelscope
LLM_AGENT_HTTP_ENDPOINT=http://your-llm-host/v1/chat/completions
LLM_AGENT_HTTP_MODEL=your-model
LLM_AGENT_API_KEY=your-api-key

SQL_RESULT_AGENT_PROVIDER=modelscope
SQL_RESULT_AGENT_HTTP_ENDPOINT=http://your-llm-host/v1/chat/completions
SQL_RESULT_AGENT_HTTP_MODEL=your-model
SQL_RESULT_AGENT_API_KEY=your-api-key
```

说明：

- `LLM_AGENT_*`：用于配置中心推荐、字段语义、场景配置相关能力。
- `SQL_RESULT_AGENT_*`：用于从业务问题生成 Query Plan、SQL、执行说明。
- 复制给别人前，不建议把真实 API Key 明文交付；让对方替换成自己的 key。
- endpoint 需要是兼容 OpenAI Chat Completions 形态的地址。

## 13. 运行数据和产物目录

运行期间产生的数据主要在：


如果只复制代码和 SQL，`runtime_data/` 可以不复制；如果要连同已生成产物一起交接，需要复制 `runtime_data/artifacts/`。

## 14. 当前业务数据边界

当前库是真实 MySQL 数据，不是前端假数据或缓存。但业务字段存在以下边界：

- 平台/店铺字段不可严肃用于平台价差：`ShopName` 为空，`ShopId` 和 `Source` 取值单一。
- 销量字段不可严肃用于真实爆款：`TotalSales` 当前基本为 0。
- 没有结构化尺码表或尺码字段，尺码只能从描述文本做候选抽取。
- 颜色、图案、材质、功能、场景、品牌、价格、品类、时间字段可用于当前四个场景分析。

因此交接后验证 SQL 质量时，必须看业务问题是否落在可用字段范围内，不能把无效字段伪造成业务结论。

## 15. 推荐交接检查清单

交给别人前按顺序检查：

1. `docs/Sql/dataservice_test_local_current_2026-05-15.sql` 已随项目一起复制。
2. 对方 MySQL 已创建并导入 `dataservice_test_local`。
3. `deploy/fz-workflow-demo.app.env` 已替换为对方自己的数据库账号和大模型 key。
4. `.venv` 已安装后端依赖。
5. `bash scripts/start_vibe_stack.sh` 能启动后端和前端。
6. `curl http://127.0.0.1:18900/api/v1/scenes` 能返回场景配置。
7. 前端 `http://127.0.0.1:18901/` 能打开，并能执行一个预设问题。
8. SQL 执行结果能保存到会话历史。
9. 汇报产物区能基于已保存的 SQL 结果生成产物。

## 16. 客户交接包和 Git 上传口径

项目本地目录较大，主要来自可重建依赖和缓存，例如 `.venv`、`node_modules`、`.next`、Presenton/DeerFlow 的虚拟环境和构建目录。

客户交接不能删除业务流程，但可以排除可重建依赖。需要保留的是：

- 当前完整数据库 SQL：`docs/Sql/dataservice_test_local_current_2026-05-15.sql`
- 配置中心、四个场景、预设问题、SQL 执行历史：已包含在当前完整 SQL 中
- 前后端源码：`apps/`、`packages/`、`integrations/`、`scripts/`
- 文档：`docs/` 和根目录 PRD/架构文档
- 已生成给客户看的产物：`runtime_data/artifacts/`、`runtime_data/presenton/app_data/exports/`
- Presenton/DeerFlow 源码：保留源码，排除虚拟环境、node_modules、构建缓存

不要上传：

- `.venv/`
- `node_modules/`
- `.next/`、`.next-build/`
- `__pycache__/`
- 日志和 PID
- `deploy/fz-workflow-demo.app.env`，因为它可能包含真实 API Key

已提供安全的配置模板：

```text
deploy/fz-workflow-demo.app.env.example
```

本地生成客户交接包：

```bash
bash scripts/package_customer_handoff.sh
```

该脚本会生成：

```text
handoff/fz_workflow_customer_handoff_YYYY-MM-DD.tar.gz
handoff/fz_workflow_customer_handoff_YYYY-MM-DD_manifest.txt
```

这套口径不会丢失预设问题和 SQL 结果，因为这些内容随 `dataservice_test_local_current_2026-05-15.sql` 一起交付；客户导入该 SQL 后，配置中心和历史记录会恢复。
