# 最终交付：一键启动与实施说明

## 1. 你醒来后会看到什么

已交付一个可运行的演示项目，覆盖以下闭环：

1. 创建分析会话（`Analysis Session`）
2. 输入自然语言目标
3. 自动生成 `Query Plan`
4. 自动生成并执行只读 SQL
5. 输出数据预览与洞察文本
6. 自动沉淀为 PPT 页面
7. 导出可编辑 `.pptx`

项目路径与关键文件：

- 演示后端入口：`demo/app/main.py`
- 执行链路引擎：`demo/app/engine.py`
- 造数脚本：`demo/app/mock_data.py`
- 前端工作台：`demo/app/static/index.html`
- 一键启动（Mac/Linux）：`scripts/one_click_start.sh`
- 一键启动（Windows）：`scripts/one_click_start.ps1`
- 方案文档（客户版+实现版）：`Vibe_Data_Analysis_客户展示版与实现方案_v0.1.md`

## 3. 使用流程（演示标准版）

1. 打开页面，点击“新建会话”。
2. 在目标输入框输入业务问题，例如：
   `请分析最近30天女装外套在不同平台和品牌的价格结构变化，并给出下一步建议。`
3. 点击“执行分析”，查看：
   - Query Plan
   - SQL
   - 洞察结论
   - 数据预览
   - 下一步推荐
4. 修改目标再执行 1-2 次，形成多页分析过程。
5. 点击“导出PPT”，下载 `.pptx`。

## 4. 每一步为什么这么做（技术 + 业务）

下面是本次交付的关键设计决策和理由。

### 4.1 先做 `Analysis Session` 再做“单问单答”

技术理由：
- 多轮分析必须有状态存储，不然后续“推荐下一页”没有上下文。
- 可追溯链路（goal, plan, sql, result, slide）需要有会话容器。

业务理由：
- 客户要的是“完成一份报告”，不是“回答一个问题”。
- 会话形态更贴近实际分析工作流。

### 4.2 SQL 生成分成 “Query Plan -> SQL”

技术理由：
- Query Plan 是 SQL 的中间表示，便于校验和调试。
- 后续接多数据源时可以保持 Planner 层稳定。

业务理由：
- 业务方可以先看分析意图是否对，再看 SQL 细节。
- 降低“黑盒模型乱写 SQL”带来的信任问题。

### 4.3 强制只读 SQL 与危险关键字拦截

技术理由：
- 演示环境和后续生产环境都需要统一安全底线。
- 先做静态拦截，后续可升级 AST 级别校验。

业务理由：
- 防止误写库或误操作。
- 客户更容易接受可控的数据访问策略。

### 4.4 先用 SQLite + Mock Data，而不是先接真实数据源

技术理由：
- 你当前目标是“睡醒就能看到完整可运行结果”，Mock 数据最快闭环。
- SQLite 部署零依赖，减少环境差异问题。

业务理由：
- 售前演示最重要是展示方法论和体验路径。
- 先让客户理解价值，再推进真实数据接入。

### 4.5 前端先用单页静态工作台

技术理由：
- 最小技术栈更稳定，更容易一键跑通。
- 把复杂度留给核心链路，而不是 UI 工程化。

业务理由：
- 演示重点在分析流程和可解释性，不在精细交互。
- 早期避免投入过多前端重构成本。

### 4.6 导出 PPT 先交付“可编辑文字版”

技术理由：
- `python-pptx` 可以稳定输出可编辑内容页。
- 先保证正确性和可编辑，再迭代视觉模板。

业务理由：
- 客户最关心“是否能直接拿去汇报”。
- 演示阶段“可编辑”比“高保真视觉”更重要。

## 5. 我对原文档的修正判断

你给的架构文档主流程和目标是正确的，但有几处需要实际修正。

### 5.1 修正点：不要一开始做太多插件化

原思路：
- 多源数据、RAG、Skills、素材检索一并考虑。

修正：
- 当前交付先锁定单数据源、单链路、单工作台。

原因：
- 否则第一版会过度设计，演示不稳定，进度不可控。

### 5.2 修正点：先演示“闭环成功率”再追求“推理复杂度”

原思路：
- 强调复杂的智能体和推荐链。

修正：
- 先保证 10 分钟内可稳定完成 1 个完整报告闭环。

原因：
- 客户感知价值来自“跑通并可复用”，不是架构图复杂度。

### 5.3 修正点：先保证可解释链路，再考虑炫技能力

原思路：
- 扩展能力很丰富。

修正：
- 每页必须可回看 goal、plan、sql、source、insight。

原因：
- 分析系统的信任建立比功能广度更关键。
