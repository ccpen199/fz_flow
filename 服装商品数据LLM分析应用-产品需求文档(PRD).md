# 服装商品数据 LLM 分析应用 — 产品需求文档（PRD）

## 1. 文档信息

- 文档名称：服装商品数据 LLM 分析应用 — 产品需求文档（PRD）
- 文档用途：用于内部需求评审、范围确认、方案立项与后续原型/设计输入
- 文档阶段：V1 评审稿
- 产品形态：Web 应用
- 当前定位：单团队 PoC / 本地原型优先

---

## 2. 项目背景

当前服装品牌在线商品数据分析通常依赖人工处理，主要流程为：从数据库或数据表中取数、手工编写 SQL、人工比对品牌和平台、整理图表、再手工制作汇报材料。该方式在实际业务中存在以下问题：

1. 分析门槛高：业务分析师了解业务问题，但不一定熟悉数据库结构，也不一定具备稳定编写 SQL 的能力。
2. 取数效率低：同一个问题往往需要多轮试错取数，且分析角度切换成本高。
3. 数据理解成本高：服装商品数据涉及平台、品牌、品类、SKU、价格、上下架、快照等多类表结构，表之间关系复杂，普通业务用户难以直接理解。
4. 结果可信度不稳定：如果让 LLM 直接读取数据库元信息生成 SQL，容易因为不了解表关系、字段语义和业务口径而生成错误查询。
5. 报告沉淀效率低：分析完成后还要再把结果搬运到 PPT 中，重复劳动多。

因此，需要建设一个将 `LLM`、`数据库查询`、`场景化建模`、`ER 关系配置` 和 `PPT 报告生成` 结合起来的数据分析应用，帮助业务分析师围绕服装品牌在线商品数据，以自然语言方式完成多步分析和报告沉淀。

---

## 3. 项目目标

本项目第一阶段目标是建设一个面向业务分析师的商品数据分析工作台，通过 LLM 与数据库结合，实现以下目标：

### 3.1 业务目标

- 降低商品分析门槛，让业务分析师通过自然语言完成分析。
- 提高分析效率，减少反复手工写 SQL、导出数据、制作图表的时间成本。
- 提升分析透明度，让用户能查看 SQL、数据来源和图表依据，增强信任度。
- 将分析结果自动沉淀为 PPT 页面，缩短从“分析”到“汇报”的路径。

### 3.2 产品目标

- 提供一个可选场景的 Web 分析工作台。
- 提供一个由 IT/数据管理员维护的场景建模能力。
- 提供基于场景语义层和 ER 关系的 SQL 生成能力。
- 提供支持多轮追问的自动分析 Agent。
- 提供“每轮分析生成一页 PPT”的持续报告能力。

### 3.3 第一阶段成功标准

若以下目标达成，则可认为第一阶段需求方向成立：

- 业务分析师可以在不写 SQL 的情况下完成典型商品分析问题。
- 系统生成的 SQL 在大部分场景下能够被用户理解并接受。
- 用户可通过多轮追问完成从初步分析到专题分析的过程。
- 每轮分析结果能够自动沉淀到 PPT 并导出 `.pptx`。
- 管理员可通过配置场景、字段和关系，显著降低 LLM 错误 SQL 的发生率。

## 4. 产品定位

本产品不是传统 BI 报表系统，也不是开放式通用聊天机器人，而是一个 `面向服装商品数据分析场景的对话式分析工作台`。

其核心定位如下：

- 对业务用户而言，它是一个“会分析商品数据、会解释结果、会持续生成汇报材料”的分析助手。
- 对数据管理员而言，它是一个“通过场景、字段和关系配置来约束 LLM 行为”的场景化建模系统。
- 对组织而言，它是一个从“问问题”到“看图表”再到“出 PPT”的一体化商品分析工具。

---

## 5. 产品范围

### 5.1 第一阶段纳入范围（In Scope）

第一阶段纳入范围如下：

- Web 应用形式的分析工作台。
- 以服装品牌在线商品数据为分析对象。
- 从最终业务库中读取数据进行只读分析。
- 支持多平台、多品牌商品数据分析。
- 支持按天历史快照分析。
- 支持场景化配置：表、字段、字段语义、ER 关系、分析指导语。
- 支持自然语言问数和自动多步分析。
- 支持结果展示：业务结论、图表、数据摘要、SQL、引用字段。
- 支持每轮分析自动生成一页 PPT。
- 支持持续追加 PPT 页面并导出可编辑 `.pptx`。
- 支持多模型接入的预留设计。

### 5.2 第一阶段不纳入范围（Out of Scope）

以下内容不纳入第一阶段：

- 爬虫采集逻辑。
- ETL/同步程序。
- 数据修复、修数、数据治理平台能力。
- 登录、权限控制、多租户。
- 向数据库写回数据。
- 执行 DDL/DML 或数据库运维类动作。
- 定时报表推送。
- 第三方 IM/办公平台集成。
- 严格企业模板级 PPT 规范复刻。

## 6. 目标用户与角色

### 6.1 核心用户：业务分析师

核心使用者为业务分析师，其典型特征如下：

- 懂业务，知道想分析什么。
- 不一定熟悉数据库结构或 SQL。
- 关注结论、趋势、对比和异常发现。
- 需要图表与汇报型输出。
- 在关键场景下希望查看 SQL，确认系统是否真正理解业务。

### 6.2 管理角色：IT / 数据管理员

系统需要有一类管理角色负责建模和配置，职责包括：

- 创建分析场景。
- 选择场景可用的表和字段。
- 配置字段中文名、别名和业务术语。
- 配置 ER 图关系和推荐 join 路径。
- 配置分析指导语和默认分析维度。
- 控制场景的数据访问边界。

### 6.3 第一阶段角色边界

虽然第一阶段不做登录能力，但需求定义中仍保留以下角色边界：

- 管理员：负责配置。
- 普通用户：负责分析和生成报告。

该边界有助于后续扩展权限系统时不推翻产品结构。

## 7. 数据范围与分析对象

### 7.1 数据来源

系统第一阶段的数据来源为“最终业务库”。

约束如下：

- 系统不关心爬虫如何采集数据。
- 系统不关心 ETL 如何将数据同步进业务库。
- 系统只负责基于业务库进行只读查询和分析。

### 7.2 数据对象

第一阶段主要分析对象为服装品牌在线商品数据，具备以下已知条件：

- 数据覆盖多平台、多品牌。
- 数据以商品相关信息为核心。
- 数据保留按天历史快照。
- 分析重点聚焦商品而非订单、会员、投放等其他域。

### 7.3 推荐纳入的数据域

为了支持第一阶段分析能力，业务库中建议至少具备以下数据域之一或其组合：

- 商品主数据
- 品牌维度
- 平台维度
- 品类维度
- 商品属性信息
- 商品价格快照
- 商品上下架状态信息
- SKU/规格信息

### 7.4 对数据侧的最低要求

为保证系统可用，数据侧至少需满足以下前提：

- 有较稳定的商品标识，能在历史快照中识别同一商品。
- 有可用于趋势分析的日期字段。
- 有品牌、平台、品类等核心分析维度。
- 表之间存在相对明确的业务关联关系。
- 历史快照口径具备基本一致性。

## 8. 核心业务场景

第一阶段需要覆盖以下三大类分析方向，且并非三选一，而是全部纳入：

### 8.1 竞品与价格分析

重点关注：

- 品牌之间的价格带分布。
- 不同平台的价格策略差异。
- 折扣力度分布。
- 历史价格变动趋势。
- 某类商品的竞品对比情况。

典型问题示例：

- 最近 30 天各品牌连衣裙的价格带分布如何？
- 哪些品牌近期在羽绒服品类的促销力度明显加大？
- 某平台中 Zara 与优衣库的外套价格结构差异是什么？

### 8.2 商品结构分析

重点关注：

- 品类结构。
- SKU 丰富度。
- 颜色/尺码结构。
- 上新结构。
- 不同品牌和平台之间的商品布局差异。

典型问题示例：

- 某品牌女装商品结构是否过于集中？
- 哪些品牌在牛仔裤品类的 SKU 最丰富？
- 各平台春季新品的颜色结构有什么差异？

### 8.3 趋势与爆款分析

重点关注：

- 上新节奏。
- 商品数量变化。
- 价格波动趋势。
- 明显异常变化。
- 潜在爆款或重点关注商品。

典型问题示例：

- 哪些品牌最近两周上新节奏明显加快？
- 哪些品牌最近 60 天价格波动最明显？
- 哪些品类在近 30 天出现异常变化？

## 9. 产品总体流程

## 9.1 管理员准备阶段

在业务用户真正使用前，需要由管理员完成前置配置。

### 步骤 1：创建分析场景

管理员创建不同分析场景，例如：

- 商品价格分析
- 竞品分析
- 商品结构分析
- 上新趋势分析

### 步骤 2：配置可用表

管理员为每个场景配置允许使用的数据表，例如：

- 商品主表
- 品牌表
- 平台表
- 价格快照表
- 商品属性表
- 品类表

### 步骤 3：配置字段语义

管理员为各表字段补充业务语义，包括：

- 字段中文名
- 业务说明
- 字段别名
- 可分组/筛选/聚合属性
- 时间字段标记
- 禁用字段标记

### 步骤 4：配置 ER 关系

管理员通过 ER 编辑器建立表与表之间的关联关系，包括：

- 关联字段
- 连接类型
- 推荐 join 路径
- 关系说明

### 步骤 5：配置分析指导语

管理员为场景维护分析提示规则，例如：

- 优先分析哪些维度。
- 不同问题优先使用哪些表。
- 模糊问题时优先拆解方式。
- 特定口径的默认解释。

### 步骤 6：发布场景

配置完成后，场景可供业务用户选择并使用。

## 9.2 业务用户分析阶段

### 步骤 1：选择分析场景

用户进入系统后先选择分析场景，系统不允许未选场景直接问数。

### 步骤 2：输入自然语言问题

用户以自然语言提出问题，例如：

- 分析最近 30 天各品牌羽绒服价格带变化
- 帮我比较 Zara 和优衣库在连衣裙上的竞品情况
- 帮我从多个维度分析近期女装新品结构变化

### 步骤 3：Agent 理解上下文并拆解任务

系统会综合以下信息进行分析：

- 当前场景配置
- 表和字段语义
- ER 关系
- 分析指导语
- 当前用户问题
- 历史对话上下文

### 步骤 4：生成并执行 SQL

Agent 自动规划分析步骤，生成 SQL，并从业务库执行只读查询。

### 步骤 5：返回结果

系统展示：

- 业务结论
- 图表
- 数据摘要
- SQL 查看入口
- 使用的表和字段
- 建议下一步分析方向

### 步骤 6：沉淀为 PPT 页面

系统将本轮有效分析自动生成一页 PPT，并追加到当前 deck。

### 步骤 7：继续追问或结束

用户可继续追加问题、改变分析方向、要求总结，直到结束本次报告。

## 9.3 报告输出阶段

- 用户可在任意时刻查看已生成的页面。
- 用户可继续在当前会话下追加页面。
- 用户结束后可导出 `.pptx`。
- 用户后续重新进入该会话时，可继续追加新页。

## 10. 功能需求

## 10.1 场景管理模块

### 10.1.1 场景基础信息管理

系统需支持管理员创建和维护场景，至少包括以下信息：

- 场景名称
- 场景描述
- 分析目标说明
- 适用范围
- 默认分析建议
- 启用状态

### 10.1.2 表管理

每个场景需支持配置可用表，至少包括：

- 表名
- 表说明
- 表用途
- 是否主表
- 是否可直接查询
- 推荐使用场景

### 10.1.3 字段管理

字段配置至少支持以下内容：

- 字段名
- 中文名
- 字段说明
- 别名/业务术语
- 字段类型
- 是否可筛选
- 是否可分组
- 是否可聚合
- 是否可排序
- 是否为时间字段
- 是否禁用

### 10.1.4 ER 关系编辑器

系统需提供可视化关系编辑器，支持：

- 表与表的连线配置
- 关联字段设置
- 连接方向和说明
- 推荐 join 路径设置
- 关系图保存

### 10.1.5 分析指导语配置

系统需允许管理员配置场景专属分析指引，帮助 Agent 理解：

- 优先分析维度
- 默认分析拆解方式
- 表使用优先级
- 常用业务术语和口径

## 10.2 分析对话模块

### 10.2.1 场景选择

用户必须先选择场景，再发起分析。

### 10.2.2 多轮会话

系统需支持完整的多轮上下文，至少保留：

- 用户提问历史
- Agent 已执行分析步骤
- SQL 执行记录
- 图表与结论历史
- 当前 PPT 页序

### 10.2.3 自动多步分析

Agent 需支持以下能力：

- 自动识别问题核心目标。
- 自动选择首轮分析维度。
- 自动生成一个或多个 SQL。
- 根据前一轮结果决定下一步方向。
- 根据用户反馈动态调整分析计划。

### 10.2.4 用户可打断和改向

用户需可在任意轮进行以下操作：

- 停止当前方向。
- 增加筛选条件。
- 切换分析维度。
- 指定对比对象。
- 要求生成总结页。

系统需保留上下文，不因为方向切换而丢失已完成工作。

### 10.2.5 结果展示

每轮分析结果应至少包含：

- 本轮核心结论
- 图表
- 支撑数据摘要
- SQL 查看入口
- 使用的表和字段
- 推荐继续分析的问题

### 10.2.6 可解释性

系统不能只给结论，必须能让用户校验结果依据，至少展示：

- 数据来源
- SQL
- 图表
- 关键指标值

## 10.3 SQL 生成与执行模块

### 10.3.1 只读约束

系统只允许执行只读 SQL，不允许任何写操作或结构修改操作。

### 10.3.2 SQL 生成依据

SQL 生成时必须优先参考：

1. 当前场景
2. 场景可用表字段
3. ER 关系配置
4. 场景指导语
5. 用户当前问题
6. 当前会话上下文

### 10.3.3 SQL 校验规则

执行前至少校验以下内容：

- 是否访问白名单外表。
- 是否访问禁用字段。
- 是否包含写操作。
- 是否存在明显高风险查询。
- 是否绕过推荐关系。
- 是否可能返回超大结果集。

### 10.3.4 查询结果标准化

查询结果需转换为统一结果结构，以供以下模块复用：

- 前端表格展示
- 图表渲染
- 业务结论生成
- PPT 页面生成
- 历史会话回看

### 10.3.5 SQL 透明度

系统默认自动执行 SQL，但用户需可查看：

- 完整 SQL
- 使用的表
- 使用的字段
- 查询结果摘要

## 10.4 图表与结果呈现模块

### 10.4.1 第一阶段图表类型

建议第一阶段支持：

- 柱状图
- 折线图
- 堆叠柱状图
- 环形图
- 排行榜
- 表格

### 10.4.2 图表选型原则

- 趋势类优先折线图。
- 对比类优先柱状图。
- 构成类优先堆叠图或环形图。
- TopN 类优先排行榜或条形图。

### 10.4.3 文案要求

结论文案应满足：

- 清楚说明主要发现。
- 引用关键数字。
- 尽量避免空泛描述。
- 在必要时提示数据限制或口径前提。

### 10.4.4 后续分析建议

每轮分析后，Agent 应能主动推荐下一步可继续分析的方向，例如：

- 按品牌拆解
- 按平台对比
- 按价格带细分
- 切换时间窗口
- 生成总结页

## 10.5 PPT 报告模块

### 10.5.1 报告模式

系统采用“对话即报告”的模式：

- 每轮有效分析生成一页 PPT。
- 分页与会话绑定。
- 新一轮分析会追加新页，而不是覆盖旧页。

### 10.5.2 第一阶段模板策略

第一阶段采用统一基础模板，不要求高度定制企业模板。

建议支持以下页面类型：

- 封面页（可选）
- 分析内容页
- 总结页

### 10.5.3 内容页结构

每页内容至少包含：

- 页面标题
- 核心结论
- 图表
- 关键数据点
- 简短业务解释

### 10.5.4 连续性要求

同一会话下，系统需维护统一的 deck，并支持：

- 逐页追加
- 历史页回看
- 后续继续追加
- 页序正确维护

### 10.5.5 导出要求

导出的 `.pptx` 需满足：

- 正常打开
- 可编辑
- 文本不是图片
- 图表和布局可读
- 页面顺序正确

## 11. 页面与信息架构建议

### 11.1 页面清单

建议第一阶段至少包含以下页面：

- 场景选择页
- 分析工作台页
- 报告预览页
- 场景管理页
- 表字段配置页
- ER 编辑页

### 11.2 分析工作台布局建议

分析工作台建议采用两栏或三栏布局：

- 左侧：会话历史 / 场景说明 / 推荐问题
- 中部：对话区 / 结论区
- 右侧：图表 / SQL / 数据来源 / 当前 PPT 预览

### 11.3 管理后台布局建议

管理后台建议至少包含：

- 场景列表
- 场景编辑
- 表管理
- 字段管理
- ER 图编辑
- 指导语维护

## 12. 非功能需求

### 12.1 性能

- 常规分析问题应在可接受时间内给出首轮结果。
- 较长分析过程需给出明确进度反馈。
- 前端在等待过程中不能表现为无响应。

### 12.2 可用性

- 界面应以业务用户理解成本低为优先。
- 用户主路径应尽量短：选场景、提问题、看结果、继续追问、导出报告。
- SQL 查看应可用但不强制干扰普通用户。

### 12.3 可维护性

- 模型调用层应支持后续接入多个 provider。
- 场景配置与分析执行逻辑应解耦。
- 图表和 PPT 生成应基于统一结构结果，避免重复实现。

### 12.4 安全性

- 数据库连接默认只读。
- 场景级白名单控制表字段。
- SQL 执行前规则校验。
- 保留分析日志和查询摘要，便于排查。

### 12.5 可扩展性

- 后续可增加更多分析场景。
- 后续可扩展订单、流量、库存、会员等数据域。
- 后续可增加登录、权限、审计等能力。

## 13. 核心用户故事

- 作为业务分析师，我希望选择一个场景后直接提问，而不需要知道底层数据库结构。
- 作为业务分析师，我希望系统不只是回答一个结果，而是能主动帮我继续拆解分析。
- 作为业务分析师，我希望看到图表和结论，也能查看 SQL 来判断结果是否可信。
- 作为业务分析师，我希望每轮分析自动生成一页 PPT，减少手工整理汇报的时间。
- 作为数据管理员，我希望通过表字段和 ER 关系配置来约束 LLM，降低错误查询。
- 作为数据管理员，我希望不同分析主题通过不同场景来控制数据范围和语义。

## 14. 验收标准

### 14.1 场景配置验收

- 管理员可以创建场景。
- 管理员可以配置表和字段。
- 管理员可以配置 ER 关系并保存。
- 场景配置能实际影响 SQL 生成和查询边界。

### 14.2 分析能力验收

- 用户选择场景后可进行自然语言问数。
- 系统可自动完成至少一轮结构化分析。
- 系统支持多轮追问与上下文继承。
- 用户改变分析方向时，不丢失历史结果。

### 14.3 可解释性验收

- 每轮结果可查看 SQL。
- 每轮结果可查看引用表和字段。
- 图表、结论、SQL 结果一致。

### 14.4 报告能力验收

- 每轮有效分析生成一页 PPT。
- 新页能持续追加到同一 deck。
- 导出的 `.pptx` 能正常打开且可编辑。
- 恢复历史会话后能继续追加新页。

## 15. 当前仍需在评审中确认的关键问题

以下事项不会阻止当前 PRD 进入评审，但建议在立项或详细设计前确认，否则会影响后续设计边界。

### 15.1 商品唯一标识与跨平台映射

当前已明确系统覆盖多平台、多品牌，但尚未完全明确：

- 同一商品在不同平台是否有统一映射关系。
- 竞品对比是按品牌/品类/价格带层面，还是按相似商品层面。
- 是否已有可复用的“同款/相似款”识别规则。

**当前默认处理方式：**
第一阶段先按 `品牌 / 平台 / 品类 / 价格带` 做竞品分析，不将“同款识别”设为前置依赖。

### 15.2 核心业务口径定义

当前尚未明确以下概念是否已有统一业务定义：

- 上新商品
- 在售商品
- 售价 / 活动价 / 吊牌价
- 折扣口径
- 爆款定义
- 异常变化定义

**当前默认处理方式：**
第一阶段由管理员在场景中补充基础口径说明，先不单独建设指标平台。

### 15.3 历史快照稳定性

虽然已确认存在按天快照，但仍建议数据侧进一步确认：

- 是否每日全量快照。
- 是否存在缺天。
- 历史记录是否可追踪同一商品。
- 商品属性变化是否覆盖历史还是保留版本。

**当前默认处理方式：**
将历史快照视为可用于趋势分析的数据来源，但正式实施前需要数据侧说明快照规则。

### 15.4 PPT 模板细则

目前只明确采用统一基础模板，但尚未明确：

- 是否需要固定封面。
- 是否需要目录页。
- 是否有固定配色、字体要求。
- 是否需要页脚和数据来源标注。

**当前默认处理方式：**
第一阶段先提供 2 到 3 种标准内容页模板，优先保证稳定输出和可编辑性。

### 15.5 会话与报告保留策略

当前已明确“可继续追加 PPT”，但尚未明确：

- 一个会话保留多久。
- 是否支持多人共用一份报告。
- 是否需要版本回溯。
- 是否允许复制报告继续分析。

**当前默认处理方式：**
第一阶段按单用户单会话逻辑设计，不做多人协作和复杂版本管理。

## 16. 默认决策清单

为保证需求可评审，本文档采用以下默认决策：

- 产品形态：Web 应用
- 用户类型：业务分析师

- 导出能力：支持可编辑 `.pptx`

- 模板策略：统一基础模板

## 17. 评审目标

本次需求评审建议重点确认以下问题：

1. 产品方向是否满足真实业务分析需求。
2. 场景化建模 + LLM SQL 分析的路径是否被接受。
3. “对话即报告”的 PPT 输出方式是否适合内部汇报方式。
4. 第一阶段范围是否合理，是否需要继续收敛。
5. 数据侧是否具备支持该产品落地的前置条件。

当以上问题被确认后，即可进入下一阶段：

- 页面原型设计
- 技术方案设计
- 数据接入约束定义
- 场景配置规范设计
